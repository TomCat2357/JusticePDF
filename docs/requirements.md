# PDFスタック整理アプリ 仕様（D&D / メイン画面 / 個別画面）
Version: 1.0  
Date: 2026-01-26  
目的: DocuWorksライクな「束（ファイル）整理」と「ページ整理」を、D&D中心のUXで実現する。LLMがこの仕様だけで実装できる粒度で定義する。

---

## 1. 用語・前提

### 1.1 オブジェクト
- **PDFファイル（File）**: 1つのPDFドキュメント。メイン画面で扱う単位。
- **ページ（Page）**: PDF内の1ページ。個別画面（ページビュー）で扱う単位。

### 1.2 画面
- **メイン画面（Main / File Grid）**
  - PDFファイル単位で、左詰めグリッドに並べる。
  - ここでは「束（ファイル）」の並び替え・統合を行う。
- **個別画面（Detail / Page Grid）**
  - メイン画面で選択したPDFを**ダブルクリック**で開く。
  - そのPDF内のページサムネイルを左詰めグリッドに並べる。
  - ここでは「ページ」の並び替え・挿入を行う。

### 1.3 D&Dの基本概念
- D&Dは **From（ドラッグ元）** と **To（ドロップ先）** の組合せで動作が決まる。
- **重ねる（Overlay / Merge）** と **挿入（Insert / Reorder）** を区別する。
- ただし **個別画面（ページ世界）では“統合”概念は持たず、常に挿入扱い**（メイン→個別も含む）。

---

## 2. 表示ルール

### 2.1 メイン画面（Main）
- PDFファイルカード（サムネ/表紙）を左詰めグリッド表示。
- 1カード = 1 PDFファイル。
- 複数選択をサポート（Ctrl/Shift、必要なら矩形選択）。

### 2.2 個別画面（Detail）
- 対象PDFのページサムネイルを左詰めグリッド表示。
- 1サムネ = 1ページ。
- 複数選択をサポート（Ctrl/Shift、必要なら矩形選択）。

---

## 3. D&D ペイロード（実装必須）

### 3.1 共通ペイロード形式（推奨）
D&Dで渡すデータは、必ず以下のいずれかに正規化する。

#### (A) Fileペイロード
```json
{
  "kind": "file",
  "files": [
    { "path": "C:/.../A.pdf" },
    { "path": "C:/.../B.pdf" }
  ]
}
````

#### (B) Pageペイロード

```json
{
  "kind": "page",
  "pages": [
    { "path": "C:/.../A.pdf", "page_index": 0 },
    { "path": "C:/.../A.pdf", "page_index": 1 }
  ]
}
```

### 3.2 複数選択の順序ルール（重要）

* **クリック順ではなく、表示順（UI上の順番）を保持**してペイロードに格納する。

  * メイン画面: グリッド上の表示順
  * 個別画面: ページ番号順（表示順）
* この順序は、挿入/統合時にそのまま使用する。

---

## 4. ドロップ位置判定（Overlay vs Insert）

### 4.1 メイン画面における判定

メイン画面の「PDFファイルカード」上にドロップする際、ドロップ座標で意味を分岐する。

* **Overlay（重ねる / merge）**

  * ドロップが対象カードの「中央ゾーン」に入る場合。
  * 結果: 統合処理を行う。
* **Insert（並び替え / insert）**

  * ドロップが対象カードの「端ゾーン」または「カード間の挿入ライン」に相当する場合。
  * 結果: 並び順変更（リストのリオーダー）を行う。

> ゾーンの具体閾値は実装で定義する（例: カード矩形の内側70%を中央ゾーン等）。
> ただし必ず視覚フィードバック（ハイライト/挿入ライン）を出す。

### 4.2 個別画面では判定不要

* 個別画面（ページビュー）では **常にInsert扱い**。
* 「中央ドロップ＝統合」の概念は存在しない。

---

## 5. 操作マトリクス（最終仕様）

### 5.1 メイン→メイン（File → File）

#### 5.1.1 Overlay（重ねる）

* From: 選択された1つ以上のPDFファイル
* To: 1つのPDFファイルカード
* 動作: **to側PDFの先頭に統合**（必須ルール）

  * 新PDFのページ順: `[from_files (表示順で連結)] + [to_file]`
* 生成物: to側PDFを更新（上書き or 新規生成は実装方針で決める。少なくとも結果がto側として見えること）
* 注意:

  * FromにTo自身が含まれる場合は無視/禁止（自己統合防止）。

#### 5.1.2 Insert（並び替え）

* From: 選択された1つ以上のPDFファイル
* To: グリッド上の挿入位置（カード間）
* 動作: **選択ブロックを保持したまま**並び替え。

  * 選択された複数ファイルは「ひとまとまり」で移動する。
  * Fromの内部順序は保持する。

---

### 5.2 個別→メイン（Page → File）

#### 5.2.1 Overlay（重ねる）

* From: 選択された1つ以上のページ（同一PDF/別PDF混在可）
* To: メイン画面のPDFファイルカード
* 動作: **to側PDFの先頭にページ統合**（必須ルール）

  * 新PDFページ順: `[from_pages(表示順)] + [to_file_pages]`

#### 5.2.2 Insert（任意位置挿入）

* From: 選択ページ
* To: メイン画面上の「カードの挿入位置」または「カード上の挿入位置UI」
* 動作: **to側PDFの指定位置に挿入**

  * どこに挿入できるかはUI設計に依存するが、最低限以下を実装する:

    * (a) 「先頭に挿入」
    * (b) 「末尾に挿入」
  * 推奨: カード上で「先頭/末尾」や簡易インジケータを用意し、挿入位置を選べるようにする。

> 重要: 本仕様では「Overlayは必ず先頭統合」。Insertは「指定位置挿入」。

---

### 5.3 メイン→個別（File → PageGrid）

* From: 選択された1つ以上のPDFファイル
* To: 個別画面（あるPDFのページグリッド）上のページ位置
* 動作: **常にInsert扱い（統合ではない）**

  * 各Fromファイルの全ページを、順に展開して挿入する。
  * 新ページ順（例）: `... [insert_pos] + [FileA pages] + [FileB pages] + ... + [rest]`
* Overlay/Insertの区別はしない。

---

### 5.4 個別→個別（Page → PageGrid）

* From: 選択された1つ以上のページ
* To: ページグリッド上の挿入位置（ページ間）
* 動作: **常にInsert扱い**

  * 同一PDF内の並び替えも、別PDF間の移動も同じ操作概念で扱う。
  * 選択された複数ページは「ひとまとまり」で移動し、内部順序は保持。

---

## 6. 複数選択D&Dの詳細挙動

### 6.1 共通

* 複数選択は常に「ブロック」として扱う（バラさない）。
* ブロック内順序は 3.2 の表示順を保持。

### 6.2 メイン画面内リオーダー

* 選択ブロックをいったん元位置から除去し、ドロップ先indexに挿入。
* 同一リスト内移動では、index補正（除去分のオフセット）を正しく処理すること。

### 6.3 ページ挿入

* 挿入先がページグリッドの場合:

  * 挿入位置indexに、選択ページ群を順序維持で挿入。
* 挿入先がメインのPDFの場合:

  * Overlay: 先頭にまとめて挿入（必須）
  * Insert: 指定位置にまとめて挿入

---

## 7. 視覚フィードバック（必須）

### 7.1 ドラッグ中表示

* 複数選択時、ドラッグゴーストに「N items」バッジを表示（N=選択数）。
* 可能なら先頭サムネ＋枚数を表示。

### 7.2 ドロップ先の可視化

* メイン画面:

  * Overlay可能: 対象カード中央をハイライト（例: “＋”）
  * Insert可能: カード間/端に挿入ライン表示
* 個別画面:

  * 常に挿入ライン表示（ページ間）

---

## 8. 禁止・例外ルール

* **自己参照禁止**

  * メイン→メイン Overlayで、FromにTo自身が含まれる場合、操作を拒否（無操作）する。
* **無効ドロップ**

  * 受け取り側が解釈できないpayload(kind不一致等)は拒否。
* **ページ番号の確定**

  * Pageペイロードのpage_indexはドラッグ開始時点のものとして扱う。
  * 同一PDF内移動では、移動後の再インデックスを更新する。

---

## 9. PDF更新のI/O方針（実装者向け）

最低限の要件:

* D&D操作後、UI上の結果が即座に反映されること（順番/内容）。
* PDFの更新は以下いずれかで実装可能:

  1. to側PDFを上書き更新
  2. 新PDFを生成し、to側として差し替え（内部でID維持）

推奨:

* 失敗時に戻せるよう、更新前にバックアップ/一時ファイル運用を行う。

---

## 10. 受け入れ基準（テスト観点）

* メイン→メイン Insertで、複数選択ブロックが順序維持で移動する。
* メイン→メイン Overlayで、to側PDFの先頭にfromが統合される。
* 個別→メイン Overlayで、to側PDF先頭にページ群が入る。
* メイン→個別で、各PDFの全ページが指定位置に順序通り挿入される（Overlay概念なし）。
* 個別→個別で、ページ群が指定位置に順序通り挿入/移動される。
* すべてのD&Dで、適切なハイライト/挿入ラインが表示される。
